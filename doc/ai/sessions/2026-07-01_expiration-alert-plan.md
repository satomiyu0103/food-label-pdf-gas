# セッション記録: 期限接近通知・期限管理シート（実装計画）

- 日付: 2026-07-01
- スコープ: 設計検討（FR-NTF-003 / FR-SHT-003 計画）

## 背景・課題

- 備蓄食料・医療品など個人利用で、消費期限が迫ったときのリマインドが必要になった
- 現行 `商品DB` は PDF 抽出の全情報（栄養成分・メモ等 25 列）を 1 シートに集約しており、期限確認の可読性が低い
- 在庫数・通知済みフラグ・消費済みステータスは PDF 由来データと性質が異なり、同一シートへの追加は運用を複雑にする
- 通知チャネルとして Slack / Gmail / Notion / Google カレンダーを比較した

## 意思決定

| 項目 | 採用 | 非採用（MVP 時点） |
|---|---|---|
| 通知チャネル | **Slack Incoming Webhook**（`notification.js` 拡張） | Gmail（OAuth 追加）、Notion（API 新規）、Google カレンダー（件数増で圧迫） |
| 通知タイミング | **毎月 1 日**（当月期限の digest）と **期限 7 日前**（個別リマインド） | 毎日一覧、期限当日のみ 等 |
| データ構成 | **`商品DB` 正本** + **`期限管理` 運用シート**（表示列は数式参照、在庫・フラグは運用シート） | 商品DB への列追加のみ、別ブック、フィルタビューのみ |
| 在庫の初期値 | PDF 1 枚処理 = **在庫 1** | — |
| 通知対象外 | `status` が消費済・廃棄、または `quantity` が 0 | — |
| 月初と 7 日前の重複 | **両方通知**（役割が異なる: 月次俯瞰 vs 直前リマインド） | どちらか一方に統合 |

### 暫定運用ルール（実装前に利用者確認可）

| 項目 | 暫定値 |
|---|---|
| 月初通知の範囲 | 期限日が**当月**に含まれる有効行を一覧 |
| 7 日前の救済 | 未通知かつ期限まで 7 日以内なら翌日以降も送信可 |
| 消費・廃棄の記録 | `status` プルダウン（有効 / 消費済 / 廃棄）と `quantity` の手動更新 |

## アーキテクチャ（計画）

```
[PDF 処理 main.js]
      │
      ├─→ 商品DB へ 1 行追記（既存 FR-SHT-001）
      └─→ 期限管理 へ 1 行ミラー（新規 FR-SHT-003）
                │
                │ 表示列: INDEX 等で商品DB 参照
                │ 運用列: quantity, status, notified_month, notified_7d
                ▼
      [日次トリガー expirationAlert.js]
                │
                ├─ 1 日 → 当月期限 digest → Slack
                └─ 毎日 → 期限 = 今日+7 日 → Slack
```

### 期限管理シート列（案）

| key | ヘッダー | 種別 |
|---|---|---|
| `inventory_row_id` | 管理ID | GAS 付与 |
| `product_db_row` | 商品DB行 | GAS 付与（リンク用） |
| （数式） | 商品名・ジャンル・期限種別・期限日・JAN | 商品DB 参照 |
| `quantity` | 在庫数 | 手入力（初期 1） |
| `status` | ステータス | 有効 / 消費済 / 廃棄 |
| `notified_month` | 月初通知済 | 例 `2026-07` |
| `notified_7d` | 7日前通知済 | 日付または TRUE |
| `notes` | 備考 | 手入力 |
