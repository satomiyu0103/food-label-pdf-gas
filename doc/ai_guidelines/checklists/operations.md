## A. 実行ログ

- [ ] バッチ開始・終了に **run_id** と件数サマリ（processed / skipped / failed）がある  
  - 根拠: [安全運用ガイド §2.1](../安全運用ガイド.md#21-ログに必ず含める項目)・[§2.5 検証](../安全運用ガイド.md#25-検証チェックリスト)
- [ ] 各処理単位で **stage** + 対象 ID がログに出る  
  - 根拠: [§2.1](../安全運用ガイド.md#21-ログに必ず含める項目)
- [ ] 失敗時の ERROR に **段階名** が含まれる（どこで落ちたか一目で分かる）  
  - ダメな例: `Error: undefined` のみ
- [ ] 秘密情報・個人情報全文・巨大ペイロード全文がログに出ない  
  - 根拠: [§2.3](../安全運用ガイド.md#23-ログに書かないもの)

## B. 例外処理と通知

- [ ] 予期可能な失敗（重複・枠不足等）と致命的失敗の応答がコード上で分かれている  
  - 根拠: [§3.1](../安全運用ガイド.md#31-例外の分類とプログラムの応答)・[§3.6 検証](../安全運用ガイド.md#36-検証チェックリスト)
- [ ] 障害通知チャネルが **1 つ** に設計上決まっている（未決なら ROADMAP に明記）  
  - 根拠: [§3.3](../安全運用ガイド.md#33-チャネル選定一般)・[03_システム設計.md](../../specs/03_システム設計.md)
- [ ] 通知本文に API キー・トークン・個人情報全文が含まれない  
  - 根拠: [§3.4](../安全運用ガイド.md#34-通知本文に含める項目)
- [ ] 連続失敗時の重複抑制がある、または「未実装」と [06_ROADMAP.md](../../specs/06_ROADMAP.md) に明記  
  - 根拠: [§3.5](../安全運用ガイド.md#35-重複抑制レート制限)

## C. 処理量・上限・再実行

- [ ] 外部 API 呼び出し前に残枠 or 上限チェックがある（無制限は specs に明示）  
  - 根拠: [§4.1](../安全運用ガイド.md#41-制限の種類)・[§4.6 検証](../安全運用ガイド.md#46-検証チェックリスト)
- [ ] 上限超過時の挙動（スキップ vs 中止）が要件・設計に書いてある  
  - 根拠: [§4.4](../安全運用ガイド.md#44-安全な打ち切り中止)・[02_要件定義.md](../../specs/02_要件定義.md)
- [ ] バッチ終了時に件数サマリと枠消費（または所要時間）がログ or 台帳に残る  
  - 根拠: [§4.2](../安全運用ガイド.md#42-監視する指標)・[§4.3](../安全運用ガイド.md#43-見積予測と実測)
- [ ] 同じ入力の再実行で二重登録・二重課金にならない（自然キー・処理済みフラグ）  
  - 根拠: [§4.5](../安全運用ガイド.md#45-冪等性と再実行)・[試験実装のエラー.md](../試験実装のエラー.md)
