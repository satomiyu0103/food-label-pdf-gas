# 実装規約（Human + Agent）

> 最終更新: 2026-06-20

## 1. 変更前確認
- 実装前に要件定義（`doc/specs/02_要件定義.md`）との差分を確認する。
- 要件外機能を見つけた場合は実装を止め、先に要件更新を提案する。

### 作業ブランチ（必須）

**コード・設定・テストの変更は、`main`（またはチーム指定のベースブランチ）上で直接行わない。** 実装着手前に、作業内容が分かるブランチを切ってから編集する。

| ルール | 内容 |
|---|---|
| タイミング | ファイル編集・コミットの**前**（要件確認の直後） |
| ベース | 最新の `main`（または指定ブランチ）から作成 |
| 命名 | 接頭辞 + kebab-case。§2 の「1 PR = 1 責務」が分かる名前にする |
| 接頭辞の例 | `feat/` 機能・フェーズ実装、`fix/` バグ修正、`docs/` ドキュメントのみ、`chore/` 設定・ツール |

命名例: `feat/phase-1-foundation`, `fix/holidays-cache-fallback`

```bash
git checkout main
git pull
git checkout -b feat/your-topic
```

緊急のホットフィックスなど、ベースブランチ上で直接作業する例外はチーム合意がある場合のみとする。

## 2. PR 粒度
- 1 PR = 1 責務
- 目安: 15〜300 行
- コミットは atomic に分割する

### コミットメッセージ（必須）

**件名・本文は日本語で書く。** AI やツールが英語のみの文案を出しても、そのまま採用しない。

| ルール | 内容 |
|---|---|
| 言語 | **日本語**（チームメンバーが読んで意図が伝わること） |
| 接頭辞 | `feat` / `fix` / `docs` / `chore` 等の Conventional Commits 形式は可 |
| 件名 | 1 行で変更の要点（目安 50 字以内）。接頭辞のあとは**日本語**で要約 |
| 本文 | 必要なら箇条書きで理由・影響範囲・関連 FR コードを補足 |

**良い例**

```
feat(phase-2): 個人希望を年間カレンダーに反映

- paid_leave_dates スキーマを追加
- 週次パターンの CUI 出力を拡張
```

**避ける例**

- `implement fiscal year calendar`（英語のみで意図が曖昧）
- `update` / `fix bug`（何をしたか不明）

## 3. 命名規約

| 対象 | 規則 |
|---|---|
| 変数・関数 | `snake_case` 日本の小中学生で習う英単語を利用する※ |
| クラス | `PascalCase` |
| 定数 | `UPPER_SNAKE_CASE` |
| 例外クラス | `〇〇Exception` サフィックスを付ける |
※直感的な理解のため。例：Force→Powerなど

## 4. テスト規約

### テスト配分
- Unit: 70%（ドメインロジック・変換処理を中心に）
- Integration: 30%（外部 API モック込みの結合フロー）

### MVP 最低ライン

<!-- プロジェクト固有のコアロジックを記入する -->
検証項目は [checklists/implementation.md](checklists/implementation.md) §C（根拠: 本節）。

### 命名規則
`test_{対象}_{期待結果}` の形式。例: `test_differ_detects_created_events`

### コーディング規約
- 新規ロジックには最低 1 件の unit test を追加する。
- 外部 API を直接叩くテストは `tests/integration/` に限定する。
- `tests/unit/` では外部 API 呼び出し禁止（モック／スタブを使う）。

### 実行コマンド
```bash
uv run pytest tests/unit -v
uv run pytest tests/integration -v
```

## 6. ログ規約

バッチ・自動化・API 連携の **構造化ログ・段階名・run_id・禁止事項** は [安全運用ガイド.md](安全運用ガイド.md) §2 を正本とする。

| レベル | 用途 |
|---|---|
| DEBUG | 開発・障害調査時（本番は通常 OFF） |
| INFO | 処理の開始・終了・件数 |
| WARNING | スキップ・軽微な異常 |
| ERROR | 失敗理由・例外メッセージ |
| FATAL | 実行全体を続行できない障害 |

**通知・処理量上限・安全な打ち切り** も同ガイド §3〜§4 を参照する。

## 7. 禁止事項
- 仕様未確定のまま実装しない。
- **`main`（ベースブランチ）上でブランチを切らずに実装・コミットしない**（§1 作業ブランチ。ホットフィックス等の例外は合意がある場合のみ）。
- 複数機能を 1 PR に混在させない。
- 既存公開 API を事前合意なく破壊しない。
- 秘密情報（APIキー・パスワード等）をコード・ログ・テストデータに書かない。

## 8. 外部 API 呼び出しのベストプラクティス

外部サービスへの呼び出しは「失敗が前提」の設計にする。

- フィルタは API 側でかける（全件取得後に手元でフィルタしない）
- リトライは指数バックオフで実装する（1s, 2s, 4s …）
- タイムアウトを必ず設定する（デフォルト無制限は禁止）
- レスポンスは型変換してドメインモデルに詰め替えてから使う

## 10. src/ README 配置ルール

### 規模の定義（メトリック）

以下のどちらか一方でも上位閾値を超えたら、その区分とみなす。

| 区分 | `src/` 直下のサブシステム数 | `src/` 全体のファイル数 |
|---|---|---|
| 小規模 | 1〜2 | 〜20 |
| 中規模 | 3〜4 | 21〜100 |
| 大規模 | 5 以上 | 101 以上 |

### README を置く予兆（トリガー条件）

以下のうち **2 つ以上** 該当したら、そのディレクトリに README を置く。検証用チェックは [checklists/implementation.md](checklists/implementation.md) §E。

- そのディレクトリの責務を一文で説明できない
- ディレクトリ内のファイル数が **10 を超えた**
- 他のサブシステムと独立した依存グラフを持つ
- 「どのファイルから読めばいいか」が外部から判断できない
- `agent_core.mdc` / `agent_implement.mdc` / `Project_map.md` の参照先に「このフォルダを読め」と書きたくなる

### 配置ルール

| ルール | 内容 |
|---|---|
| 置く単位 | サブシステム直下のみ（例: `src/backend/`, `src/frontend/`） |
| 置かない場所 | 末端ディレクトリ（例: `src/backend/api/routes/`） |
| セットで行う作業 | `Project_map.md` の参照先早引き表に README へのリンクを追加する |

### README の最小記述項目

```markdown

## 入口ファイル
（最初に読むべきファイル名）

- 依存先:
- 依存元:

## ローカル実行
（起動・テストコマンドがあれば）
```

## 12. コードコメント（意図の明示）

### 目的

初学者や AI がコードの意図（目的、前提、過去のエラーによる対処など）を把握しやすくし、理解と協力のコストを下げる。

### 背景

AI 生成コードは動くが部分最適になりやすく、作成者以外が読み解く負荷が高い。何をしているかだけでなく、**なぜそうしているか**を残す。

### 記載すること

| 種類 | 例 |
|---|---|
| **意図（Why）** | なぜこの手法・分岐・定数値を選んだか |
| **処理内容（What）** | ブロックや関数が入力から出力まで何をしているか |
| **前提・制約** | データ形式、外部 API の仕様、再現性のための固定値 |
| **過去の対処** | 既知バグ・403 等の回避策と、その根拠（Issue / 試験実装のエラーへの参照可） |

### 書き方

- **関数・クラス**: docstring に目的（1 文）と、非自明な処理の要点を書く
- **ブロック**: 分岐・ループ・例外処理・ワークアラウンドの直前に 1 行コメント
- **自明なコード**（`import`、getter、1 行の代入など）はコメント不要
- コメントは**日本語**で、コードと同じ粒度で更新する（古い意図を残さない）

    path = client.download("dataset-id", output_dir=str(output_dir))
