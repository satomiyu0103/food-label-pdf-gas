# 試験実装のエラー記録

## 目的

実装中に発生したエラーとその対策を記録する。着手前に必読。

「すべてのエラー」を書く帳ではない。**同じ症状が再発しうるもの**だけを残し、人と AI の認知負荷を下げるためのナレッジとする。

---

## 記載する条件（再利用価値のライン）

次のいずれかに当てはまるときは **記載する**。

| 条件 | 説明 |
|---|---|
| 再現しうる | 環境・手順・設定を変えると同じ症状が出る |
| 原因が非自明 | ログ・スタックトレースだけでは直感で分からなかった |
| 調査に時間がかかった | 目安: 15 分以上かかった、または複数の切り分けが必要だった |
| プロジェクト固有 | このリポジトリの構成・依存・規約に紐づく（汎用ドキュメントに載っていない） |
| 再発防止できる | ルール・チェック・テンプレ修正など、**次に防ぐ具体策**を書ける |

### 記載しない（または Issue だけで足りる）

| 対象 | 理由 |
|---|---|
| タイポ・未保存・パス打ち間違い | 再利用価値がほぼない |
| すぐ直った単純な構文・import 漏れ | エディタや linter で防げる |
| 個人 PC だけの一時障害（再起動で解消） | 他メンバーに再現しない |
| 秘密情報の混入 | 値は書かず、`.gitignore` や運用で防ぐ（ログに鍵を残さない） |

迷ったときは **「1 週間後の自分／他の開発者が同じハマりをするか？」** で判断する。Yes なら記載、No なら省略でよい。

### 他ドキュメントとの役割

| 書く場所 | 向き |
|---|---|
| **このファイル** | 症状・原因・対策・再発防止（ナレッジ） |
| `doc/specs/07_CHANGELOG.md` | リリース単位で「何を直したか」（`Fixed` + FR コード） |
| GitHub Issue / PR | 再現手順・議論・レビュー。長い調査はここに詳細を書き、このファイルには要約とリンク |

---

## エラー記録フォーマット

```
### [YYYY-MM-DD] エラーの概要（1行）

**症状**: 何が起きたか
**原因**: なぜ起きたか
**対策**: どう解決したか
**再発防止**: 同じミスを防ぐために何をするか
```
