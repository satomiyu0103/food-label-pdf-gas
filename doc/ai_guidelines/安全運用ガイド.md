# 安全運用ガイド（ログ・通知・処理量）

最終更新: 2026-05-30

> **正本の位置づけ**: 言語・ランタイム（GAS / Python / Node 等）に依存しない、**バッチ・自動化・API 連携を安全に運用するための一般原則**。  
> 本プロジェクト固有の実装（モジュール名・設定キー・toast 等）は `doc/specs/03_システム設計.md` と `src/{{APP_DIR}}/README.md` に書く。ここには書かない。

---

## 1. 目的と原則

### 1.1 何を防ぐか

| リスク | 典型例 |
|--------|--------|
| 気づかない失敗 | エラーがログだけで、担当者に届かない |
| 連鎖障害 | 1 件の失敗でバッチ全体・下流 DB を壊す |
| コスト・枠の使い切り | API 上限・実行時間・課金の超過 |
| 再実行不能 | 中途半端な状態のまま止まり、手作業復旧が必要 |
| 情報漏洩 | ログ・通知本文に秘密情報・個人情報が載る |

### 1.2 設計原則（優先順）

1. **観測可能（Observable）** — 誰が・いつ・何を・どの段階で処理したか追える。
2. **境界付き（Bounded）** — 1 回の実行に上限（件数・時間・外部 API 回数）がある。
3. **失敗を局所化（Fail partial, not catastrophic）** — 1 件失敗で全体を落とさない（方針は要件で明示）。
4. **安全な縮退（Degrade safely）** — 枠不足・依存障害時は **打ち切り・スキップ・再試行可能な残し** を選ぶ。
5. **人に届く（Actionable alerts）** — 復旧に必要な情報だけを、適切な深刻度で通知する。

## 2. 実行ログ（処理時）

### 2.1 ログに必ず含める項目

| 項目 | 説明 |
|------|------|
| **相関 ID（run_id）** | 1 回のバッチ実行を貫く UUID 等。全ログ行に付与 |
| **段階（stage）** | 取込 / 変換 / 外部 API / 永続化 など、パイプライン上の位置 |
| **対象 ID** | ファイル名・レコード ID・ジョブ ID（ユーザー向けに特定可能な粒度） |
| **結果** | 成功 / スキップ（理由コード）/ 失敗 |
| **時刻** | UTC または明示したタイムゾーン（ISO 8601 推奨） |
| **件数サマリ** | バッチ開始・終了時に processed / skipped / failed の集計 |

### 2.2 ログレベルの使い分け

| レベル | 用途 | 例 |
|--------|------|-----|
| **DEBUG** | 開発・障害調査時のみ（本番は通常 OFF） | ペイロード要約（秘密除く） |
| **INFO** | 正常系の節目 | 開始・終了・件数・段階の完了 |
| **WARNING** | 処理は続行可能な異常 | 重複スキップ、リトライ成功、枠ギリギリ |
| **ERROR** | 当該単位の失敗 | 例外メッセージ・HTTP ステータス・段階名 |
| **FATAL** | 実行全体を続行できない | 設定欠落・認証不可・データストア到達不能 |

### 2.3 ログに書かないもの

- API キー・トークン・パスワード・Cookie
- 個人情報の全文（必要なら ID のみ・マスク）
- 巨大ペイロードの全文（要約・ハッシュ・バイト数に留める）

### 2.4 構造化と保存

- **構造化ログ**（JSON 1 行 / キー付き）を推奨。後から検索・集計しやすい。
- **保存先は 2 層**が一般的:
  - **実行時**: プラットフォームのログ（Cloud Logging、stdout、ファイル）
  - **業務向け**: DB・スプレッドシート・監査テーブル（「誰がいつ何を処理したか」の台帳）
- 保持期間・ローテーションを決める（無期限はコストと漏洩リスク）。

### 2.5 検証チェックリスト

定期監査・プロトタイプ後点検は [checklists/operations.md](checklists/operations.md) §A（根拠: 本節 §2.1〜§2.3）。

## 3. 例外処理と通知

### 3.1 例外の分類とプログラムの応答

| 種別 | 説明 | 推奨応答 |
|------|------|----------|
| **予期可能** | 重複・バリデーション・枠不足 | スキップ + WARNING ログ。バッチは継続 |
| **一時的** | レート制限・タイムアウト・5xx | 指数バックオフでリトライ（上限回数あり） |
| **永続的** | 認証失敗・スキーマ不一致・404 | 当該単位を失敗として記録。通知は方針に従う |
| **致命的** | 設定欠落・DB 接続不可 | バッチ全体を中止（FATAL）+ 即時通知 |

**原則**: 「例外を握りつぶさない」≠「すべてでバッチを止める」。**要件で**「1 件失敗時に全体を止めるか」を決める。

### 3.2 通知の深刻度とチャネル

| 深刻度 | 誰が見るか | 典型チャネル | 頻度制御 |
|--------|------------|--------------|----------|
| **情報** | 実行者・運用者 | UI トースト、画面サマリ、日次ダイジェストメール | 毎回可 |
| **警告** | 運用者 | メール、チャット（Slack / Discord / Teams） | しきい値・集約 |
| **障害** | オンコール担当 | メール + チャット、PagerDuty 等 | **重複抑制必須** |
| **致命的** | オンコール + 関係者 | 上記 + 複数チャネル | 即時・エスカレーション |

### 3.3 チャネル選定（一般）

プロジェクトで **ADR または `doc/specs/03_システム設計.md` に 1 本決める**。複数チャネルに同時送信しない（ノイズと設定漏れを防ぐ）。

| チャネル | 向くケース | 注意 |
|----------|------------|------|
| **アプリ内 UI**（トースト・モーダル） | 手動実行・操作者が画面を見ている | 離席・無人運用には不向き |
| **メール** | 個人・小規模、監査 trail が欲しい | ノイズ、到達遅延 |
| **Slack / Discord / Teams（Webhook）** | チーム共有・モバイル | Webhook URL は秘密。チャネル設計 |
| **SMS / 電話** | 本番障害のみ（コスト高） | 通常は最終段階 |

**選定の目安**

- 操作者が常に画面にいる → **UI サマリ + ログ** で足りることが多い。
- 無人・夜間・定期実行 → **障害時のみ** チャット or メール。
- コンプライアンス・監査 → **構造化ログ + 業務台帳** を正本、通知は補助。

### 3.4 通知本文に含める項目

- run_id、環境（prod / staging）、処理名
- 失敗段階、対象 ID、エラー要約（スタック全文はログへリンク）
- 件数サマリ（processed / skipped / failed）
- **次のアクション**（再実行手順・ダッシュボード URL）

含めない: 秘密情報、個人情報全文、巨大スタックの貼り付け（ログ参照を促す）。

### 3.5 重複抑制・レート制限

- 同一障害は **5〜15 分に 1 通** などウィンドウでまとめる（fingerprint: エラーコード + 段階）。
- リトライ中は通知しない（最終失敗時のみ）。
- 通知送信自体が失敗しても **本処理を落とさない**（通知はベストエフォート）。

## 4. 処理量の調整（監視・見積・安全な中止）

### 4.1 制限の種類

| 制限 | 単位の例 | 超過時の一般パターン |
|------|----------|----------------------|
| **外部 API クォータ** | 回/分、回/日 | 事前チェック → スキップ or キュー残し |
| **実行時間** | 秒/実行（サーバレス上限） | チェックポイント・次回再開 |
| **メモリ・ファイルサイズ** | MB/ファイル | 拒否 + 明確なエラー |
| **同時実行** | 並列数 | キュー・直列化 |
| **ビジネス上限** | 1 日の処理件数 | 設定で cap |

### 4.2 監視する指標

| 指標 | 用途 |
|------|------|
| キュー深さ / 未処理件数 | バックログの蓄積 |
| 処理スループット（件/分） | 性能劣化の検知 |
| スキップ率・失敗率 | 品質・入力データの問題 |
| 外部 API 残枠（日次・分次） | 打ち切り判断 |
| 実行時間の p95 | タイムアウト設計の見直し |

ダッシュボードがなくても、**バッチ終了サマリ 1 行**（件数・枠消費・所要時間）をログ or 台帳に残す。

### 4.3 見積（予測）と実測

| タイミング | 内容 |
|------------|------|
| **実行前** | 対象件数、1 件あたりの外部呼び出し回数、予想 API 消費、残枠との比較 |
| **実行後** | 実際の processed / skipped / failed、API 消費、所要時間 |

**原則**: 見積と実測の差が大きいときは、入力件数の数え方・重複・リトライ・隠れ API 呼び出しを疑う。差分は開発日誌 or 運用記録に 1 行残す。

```
見積: 対象 10 件 × API 1 回 = 10 回、残枠 5 → 5 件処理・5 件翌日が妥当
実測: 処理 4 / スキップ 1 / 枠不足 5 / 所要 8 分
```

### 4.4 安全な打ち切り・中止

| 方針 | いつ使う | 挙動 |
|------|----------|------|
| **スキップして継続** | 枠不足・重複・軽微エラー | 対象だけ飛ばし、残りと未処理キューを保持 |
| **グレースフル停止** | 実行時間上限が近い | 現在単位まで完了し、チェックポイント保存 |
| **即時中止** | データ破壊リスク・認証失敗 | 以降の処理を止め、FATAL 通知 |

**避けるべき**: 枠不足で **未処理データを「処理済み」にする**、部分書き込みのまま黙って終了する。

### 4.5 冪等性と再実行

- 同じ入力を再実行しても **二重登録・二重課金** にならない設計（自然キー・処理済みフラグ）。
- 打ち切り後の **再開手順** を運用ドキュメントに書く（「INPUT に残る」「キューから再デキュー」等はプロジェクト固有なので specs 側）。
